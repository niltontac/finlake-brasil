## Summary

Camada Gold do domínio BCB entregue com 2 modelos dbt materializados no schema
`gold_bcb` do PostgreSQL existente. `macro_mensal` (grain mensal) é o SSOT analítico:
cruza SELIC × IPCA × PTAX via 3-way join com CTE + LAG, produzindo `selic_real`
(SELIC acima da inflação), `ptax_media` e `ptax_variacao_mensal_pct`. `macro_diario`
(grain diário) é derivado via `ref('macro_mensal')` + carry forward do `acumulado_12m`
para cada dia útil do mês — sem recalcular nada, toda lógica centralizada no mensal.
Macro `generate_schema_name` instalada em `transform/macros/` para prevenir concatenação
de schema (`silver_bcb_gold_bcb` → `gold_bcb`). DAG `dag_gold_bcb` orquestra via
`ExternalTaskSensor` aguardando `dag_silver_bcb` antes do `dbt run --select macro_mensal
macro_diario`. Schema `gold_bcb` provisionado via migration `003_gold_bcb.sql` idempotente.
Arquitetura PostgreSQL (vs DuckDB proposto inicialmente) foi a decisão crítica:
complexidade de engine separada não defensável para ~7k registros no contexto do portfólio.

## Lessons Learned

### Process

- **A decisão DuckDB → PostgreSQL no Brainstorm salvou a feature inteira.**
  O Brainstorm inicial foi aprovado com DuckDB. A reversão antes do /define eliminou
  PRE-04 (bind mount `./data`), `dbt-duckdb` como nova dependência, `transform_gold/`
  separado, driver JAR Metabase e a complexidade de `ATTACH`. O custo de rever uma
  decisão no Brainstorm é zero; no Build seria um retrabalho de vários arquivos.

- **Grounding arquitetural > coerência com CLAUDE.md genérico.**
  CLAUDE.md declara DuckDB para Gold como diretriz geral. A instância específica
  (BCB, ~7k records, Metabase nativo) justificou exceção documentada. ADR-1 captura
  o raciocínio: a diretriz volta a ser relevante quando CVM Gold existir com volume
  que justifique a separação de engines.

### Technical

- **`generate_schema_name` macro é obrigatória em qualquer projeto dbt multi-schema.**
  Sem ela, `+schema: gold_bcb` no `dbt_project.yml` produz `silver_bcb_gold_bcb` —
  comportamento silencioso que só aparece ao verificar onde as tabelas foram criadas.
  A macro vai em `transform/macros/` uma vez e funciona para todos os schemas futuros
  do projeto (quando CVM Gold existir, não precisará ser refeita).

- **CTE padrão para agregar + window function é necessário no PostgreSQL.**
  PostgreSQL não permite window functions diretamente dentro de funções de agregação.
  O CTE com GROUP BY primeiro e LAG na query externa é o padrão correto — o mesmo
  pattern usado no `ipca_monthly.sql` da Silver com `row_number()`. Consistência
  arquitetural entre Silver e Gold facilita onboarding.

- **`ref()` vs `source()` em Gold define corretamente o lineage dbt.**
  `ref('selic_daily')` em `macro_mensal.sql` aponta para o modelo Silver — não para
  a source Bronze. Isso garante que Gold sempre leia dados já transformados e que
  `dbt run --select +macro_mensal` suba automaticamente Silver antes de Gold quando
  necessário.

- **`--select macro_mensal macro_diario` vs `--select domain_bcb` não é detalhe.**
  O selector errado re-executaria todos os modelos Silver desnecessariamente a cada
  run Gold, aumentando tempo de execução e acoplando responsabilidades de DAGs distintas.
  O selector correto documenta explicitamente a responsabilidade do `dag_gold_bcb`.

### Tools & Libraries

- **Macro Jinja do dbt é código de infraestrutura, não boilerplate.**
  A macro `generate_schema_name` tem 8 linhas e resolve um problema crítico de
  configuração que causaria falha silenciosa em produção. Vale o PR comment explicando
  o motivo da macro existir — AT-009 valida exatamente isso.

- **`dbt compile` antes de `dbt run` é smoke test gratuito.**
  Valida Jinja, refs e SQL syntax sem tocar no banco. Útil para CI (GitHub Actions
  com `dbt compile` sem conexão ao banco) e para validar `generate_schema_name`
  resolveu o schema corretamente antes de executar.
