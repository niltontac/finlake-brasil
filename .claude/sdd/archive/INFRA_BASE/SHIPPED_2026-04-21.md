## Summary

Infraestrutura local containerizada do FinLake Brasil entregue com Docker Compose modular (Profiles + `include`), cobrindo PostgreSQL 15, Airflow 2.10.4 (standalone), Metabase e DuckDB via bind mount. A arquitetura permite subir subconjuntos de serviços por contexto (`core` para CI/CD, `orchestration` para desenvolvimento de pipelines, `full` para demo completa), alinhada com os princípios de isolamento do Data Mesh. O ambiente está pronto para receber as primeiras DAGs dos domínios BCB (Macro) e CVM (Fundos).

## Lessons Learned

### Process

- **SDD acelerou decisões críticas.** A fase de Brainstorm forçou decisões sobre LocalExecutor vs. CeleryExecutor e volumes híbridos antes de qualquer código — evitou retrabalho posterior.
- **YAGNI bem aplicado:** LangFuse, pgAdmin e Redis foram removidos no Brainstorm. A tentação de "já que vou configurar..." é real em infraestrutura; ter o processo formal de YAGNI ajudou a resistir.

### Technical

- **`docker compose config` como validação rápida:** Roda em milissegundos, valida YAML, interpola variáveis e detecta referências circulares sem subir nenhum container. Excelente gate de CI.
- **`data/` no `.gitignore` requeria ajuste:** `data/` (trailing slash) ignora o diretório inteiro, incluindo `.gitkeep`. O padrão correto é `data/*` + `!data/.gitkeep`. Identificado e corrigido durante o build.
- **Airflow `standalone` simplifica muito:** Um container em vez de cinco é decisivo para DX local e CI/CD. O trade-off (não recomendado para produção) é aceitável e documentado nos ADRs.
- **Assumption A-003 validada implicitamente:** Usar `pip` + constraints do Airflow evitou o risco de incompatibilidade com `uv`. A migração para `uv` permanece como melhoria futura.

### Communication

- **Escolha de profiles foi fácil com opções claras:** A pergunta múltipla escolha sobre volumes (opção c — híbrido) resolveu em segundos o que poderia ter sido uma discussão longa.
- **Confirmar estrutura de arquivos antes de gerar o DESIGN** (segundo checkpoint de validação) evitou discordâncias sobre onde os arquivos deveriam ficar.

### Tools & Libraries

- **Docker Compose `include` + Profiles nativos:** Elimina necessidade de scripts shell para orquestrar múltiplos arquivos `-f`. Menos código de infraestrutura, mais declarativo.
- **Makefile com `-include .env; export`:** Padrão elegante que carrega variáveis do `.env` sem falhar quando o arquivo não existe (prefixo `-`).
- **`pg_isready` como healthcheck do PostgreSQL:** Mais confiável que `CMD-SHELL pg_isready` com parâmetros; confirma que o servidor aceita conexões no database específico.
