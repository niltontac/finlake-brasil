## Lições Aprendidas

### O que funcionou bem

**1. Validação pré-design pagou dividendo imediato**
O DEFINE assumiu `sit = 'EM LIQUIDAÇÃO'` sem consultar os dados reais. A validação pré-design (`SELECT DISTINCT sit FROM bronze_cvm.cadastro`) revelou o valor correto (`'LIQUIDAÇÃO'`) antes de qualquer linha de código ser escrita. Evitou retrabalho em 4+ arquivos.

**Regra para o futuro:** Para filtros baseados em valores de enumeração, sempre validar contra os dados reais antes de fechar o DEFINE.

---

**2. Correção de coluna durante o build não atrasou**
O usuário identificou que `taxa_adm`/`taxa_perfm` já eram `NUMERIC` no Bronze (não `TEXT` como o DESIGN assumiu). A correção de `nullif(trim(inf_taxa_adm), '')::numeric` para `taxa_adm::numeric(10,4)` foi aplicada em 30 segundos — sem impacto no build.

**Regra para o futuro:** O DESIGN deve inspecionar o DDL do Bronze (`004_bronze_cvm.sql`) antes de assumir tipos de coluna.

---

**3. dbt 1.11 tem nova sintaxe YAML que vale documentar**

Três padrões novos no dbt 1.11 vs. versões anteriores:
- `tests:` → `data_tests:` (deprecado, mas ainda funciona)
- `accepted_values` e `relationships`: argumentos vão sob `arguments:`
- `severity:` vai sob `config:` dentro do teste

Exemplo correto para dbt 1.11:
```yaml
data_tests:
  - accepted_values:
      arguments:
        values: ['EM FUNCIONAMENTO NORMAL', 'LIQUIDAÇÃO']
  - relationships:
      arguments:
        to: ref('fundos')
        field: cnpj_fundo
      config:
        severity: warn
```

---

**4. `incremental_strategy: 'delete+insert'` funciona como esperado no dbt-postgres**

A Assumption A-002 do DEFINE foi confirmada: `delete+insert` disponível no dbt-postgres 1.10.0. Comportamento correto — deleta e reinsere pela `unique_key` composta, sem duplicatas.

---

### O que melhorar

**1. Range de row count no Success Criterion era assumption não-validada**

O DEFINE especificou `1.500-5.000 rows` para `silver_cvm.fundos` sem consultar o cadastro Bronze real. O dado real são 130 fundos operacionais. O range era baseado em estimates gerais do mercado, não nos dados disponíveis localmente.

**Regra para o futuro:** Success Criteria numéricas de row count devem ser derivadas de queries no Bronze, não de estimates de mercado. Adicionar uma seção "Validações de dados pré-DEFINE" no workflow.

---

**2. AT-005 (ExternalTaskSensor) não testado em ambiente de produção**

O teste de integração da DAG com o Airflow real ficou pendente (requer `dag_bronze_cvm_cadastro` ativa). O build validou sintaxe Python e lógica, mas o comportamento do sensor em execução real é um ponto aberto.

**Regra para o futuro:** DAGs novas devem ter um checklist de validação no Airflow UI antes do ship (trigger manual, verificação de sensor).
