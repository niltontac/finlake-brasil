## Summary

Ciclo Medallion do domínio BCB fechado: Bronze → Silver → Gold → Visualização.
Volume Docker nomeado `metabase-data` adicionado ao `docker-compose.yml` raiz para
persistência do H2 entre restarts — gap crítico identificado no Define que não estava
no Brainstorm. Script `scripts/export_metabase.sh` autentica via `POST /api/session`,
busca o dashboard "BCB Macro" pelo nome via `GET /api/dashboard` e exporta o JSON
completo via `GET /api/dashboard/:id`, salvando em `docs/metabase/dashboard_bcb_macro.json`.
`make metabase-export` sourcia `.env` antes de executar o script (ajuste feito pelo
usuário: `set -a && . ./.env && set +a`). `docs/metabase/SETUP.md` documenta os
valores de conexão com ênfase em `host=postgres` (não `localhost`). Setup manual
(wizard, conexão, dashboard) é executado pelo usuário após aplicar o volume.

## Lessons Learned

### Process

- **Gap de PRE detectado no Define, não no Brainstorm — validação correta.**
  O Brainstorm não capturou que `compose.metabase.yml` não tinha volume para H2.
  O Define leu o arquivo e identificou o gap antes de especificar os goals.
  Lição: o Define é o momento de ler o código existente, não apenas extrair o Brainstorm.

- **Feature de visualização tem split code/manual — Build Report reflete isso.**
  6 arquivos de código, mas os 8 ATs dependem de ação manual (wizard + dashboard).
  O BUILD_REPORT documentou explicitamente o que é "pendente de setup manual" vs
  o que está validado. Esse split é honesto e reproduzível — evita promessas de AT
  que não podem ser verificadas automaticamente.

### Technical

- **Volume em `docker-compose.yml` raiz, não no arquivo `include`d.**
  O padrão do projeto centraliza volumes globais no `docker-compose.yml` raiz
  (onde `postgres-data` já existe). Declarar `metabase-data` em `compose.metabase.yml`
  funcionaria, mas quebraria o padrão. Conhecer a estrutura `include:` do Compose
  antes de escolher onde declarar o volume evitou inconsistência.

- **`set -a && . ./.env && set +a` no Makefile é o padrão correto para sourciar `.env`.**
  O Makefile já tem `-include .env; export` no topo, mas variáveis definidas após
  a invocação (`METABASE_ADMIN_EMAIL` adicionado pelo usuário pós-wizard) podem não
  estar presentes no ambiente do make. O `set -a` garante que todas as vars do `.env`
  sejam exportadas antes de chamar o script bash. Ajuste feito pelo usuário durante
  o build — incorporado como lição.

- **Busca de dashboard por nome é mais robusta que por ID fixo.**
  ID `1` assume que "BCB Macro" é o primeiro dashboard criado. O wizard do Metabase
  pode criar dashboards de exemplo em algumas versões. Busca por nome com erro
  informativo (`Disponíveis: [...]`) é mais defensiva e auto-documentada.

- **`set -euo pipefail` + `:?` para vars obrigatórias é o mínimo de robustez em scripts de integração.**
  O `:?` no bash (`${VAR:?mensagem}`) garante que o script falha imediatamente com
  mensagem clara se `METABASE_ADMIN_EMAIL` ou `METABASE_ADMIN_PASSWORD` não estiverem
  definidas — em vez de fazer uma chamada à API com credenciais vazias e falhar com
  erro críptico de autenticação.

### Tools

- **Metabase API v1 é estável para operações básicas (session, dashboard list, get).**
  `POST /api/session`, `GET /api/dashboard`, `GET /api/dashboard/:id` funcionam desde
  Metabase 0.40+. Para export nativo com todos os metadados (incluindo configurações
  de visualização), verificar se a versão instalada tem `/api/dashboard/:id/export`
  (disponível a partir de 0.50+) — pode gerar JSON mais completo para import.
