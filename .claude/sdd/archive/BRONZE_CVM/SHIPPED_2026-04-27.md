## Summary

Camada Bronze do domínio CVM (domain_funds) implementada com sucesso. O pipeline
ingere dois datasets públicos da CVM via portal de dados abertos:

1. **Cadastro de fundos** (`cad_fi.csv`): ~30k registros com 40 colunas cadastrais,
   SCD Tipo 1, atualizado diariamente via DAG Airflow.
2. **Informe diário** (`inf_diario_fi_YYYYMM.zip`): dezenas de milhões de linhas desde
   2000, particionado por ano no PostgreSQL, com carga histórica via PySpark e delta
   mensal via Airflow.

O schema `bronze_cvm` é isolado do `bronze_bcb` (princípio Data Mesh), com idempotência
garantida por chaves compostas e `ON CONFLICT DO NOTHING/DO UPDATE`.

## Lessons Learned

### O que funcionou bem

**1. Vetorização pandas como decisão de build**
O DESIGN mencionava `iterrows()` como pseudocódigo, mas a anotação do `/build`
forçou a revisão antes de implementar. O resultado foi `validate_informe_rows()`
100% vetorizada desde o primeiro commit — não foi necessário refatorar depois.
Para CSVs de ~48MB, a diferença é de ~30s para <1s.

**Why this matters:** Quando o DESIGN usa código de referência (não production-ready),
incluir uma nota explícita de performance na invocação do `/build` evita que o agente
copie o pseudocódigo verbatim.

**2. DDL com 40 colunas — antecipação no `/design`**
A instrução explícita de incluir as 40 colunas do `cad_fi.csv` no `/design` evitou
uma refatoração dolorosa do DDL e do dict `_CSV_TO_DB` do loader. Descobrir 18 colunas
faltantes depois de testar com dados reais seria custoso.

**3. `pytest.importorskip("airflow")` para testes condicionais**
Padrão estabelecido no BCB e replicado no CVM. Os testes de loader são ignorados
automaticamente localmente, mas executam no container Airflow. Zero fricção no CI local.

**4. Bifurcação de URL encapsulada em função testável**
`build_informe_url(year, month)` centraliza a lógica HIST/DADOS em um único ponto,
testada com 6 casos. Qualquer mudança futura na URL da CVM tem um único lugar de correção.

### O que pode melhorar

**1. Verificação A-003 (ON CONFLICT em tabela particionada) deveria ser AT manual no início do build**
A assumption A-003 (PostgreSQL 15 suporta `ON CONFLICT` em tabela particionada com PK
incluindo coluna de partição) foi assumida como verdadeira sem teste prévio. O BUILD
poderia ter iniciado rodando a migration e testando um insert conflitante antes de
escrever o código dos loaders.

**2. BRAINSTORM não mencionou o diretório `scripts/spark/` como ausente**
O diretório não existia, o que forçou um `mkdir -p` adicional durante o build. Uma
checagem do estado do repo no BRAINSTORM teria identificado isso antecipadamente.

**3. Separação entre testes unitários e ITs de integração**
Os testes Airflow-dependentes são marcados com `importorskip`, mas não há uma suíte
clara de integration tests que rode no container. Um target `make test-cvm` que execute
dentro do container Airflow seria valioso para AT-001 a AT-006.

### Padrões para reutilizar em Silver_CVM e Gold_CVM

- `ON CONFLICT (cnpj_fundo) DO UPDATE SET ...` com lista dinâmica de colunas
- `pd.to_datetime(errors="coerce")` + `.notna()` para validação vetorizada de datas
- `_safe_float`/`_safe_int` para campos numéricos opcionais de fontes abertas
- `pytest.importorskip("airflow")` para isolamento de dependências de runtime
- `set -a && . ./.env && set +a` no Makefile para passar variáveis ao spark-submit
