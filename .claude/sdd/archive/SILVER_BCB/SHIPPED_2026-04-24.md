## Summary

Camada Silver do domínio BCB entregue com setup greenfield do dbt-core no FinLake Brasil.
Projeto dbt `finlake` inicializado em `transform/` com 3 modelos materializados como `table`:
`selic_daily` (taxa anualizada SELIC via convenção BCB de 252 d.u./ano), `ipca_monthly`
(acumulado 12 meses via `EXP(SUM(LN()))`) e `ptax_daily` (variação diária via `LAG`).
DAG `dag_silver_bcb` orquestra `dbt run` via `BashOperator` após aguardar `dag_bronze_bcb`
com `ExternalTaskSensor` (mode=reschedule) — downstream conhece upstream, direção Data Mesh
preservada. Schema `silver_bcb` provisionado via migration SQL idempotente. Infraestrutura
Airflow atualizada com bind mount `./transform:/opt/airflow/transform` e injeção de
`POSTGRES_USER`/`POSTGRES_PASSWORD` para o `env_var()` do dbt. 4 erros ruff pré-existentes
do Bronze corrigidos como parte do build.

## Lessons Learned

### Process

- **CTE com `row_number()` para controle de janela incompleta é o padrão correto no dbt.**
  A alternativa de usar `count(*) over (ROWS BETWEEN 11 PRECEDING AND CURRENT ROW) < 12`
  também funcionaria em PostgreSQL, mas o CTE é mais legível: torna explícito que os
  primeiros 11 meses são o caso especial, e o leitor não precisa raciocinar sobre o
  comportamento do frame da window function.

- **`--profiles-dir` flag na BashOperator é mais explícito que `DBT_PROFILES_DIR`.**
  Ao auditar o log do Airflow, o comando completo aparece no output — qualquer pessoa
  consegue entender onde o dbt busca o profiles.yml sem precisar inspecionar env vars.

- **Grounding com dados reais da Bronze no Brainstorm foi decisivo para as fórmulas.**
  `valor = 0.054266` significa % a.d. — sem isso, a fórmula incorreta `(1 + valor)^252`
  teria passado para o DESIGN e gerado dados errados silenciosamente. O erro seria
  descoberto apenas ao comparar com a meta SELIC (~14.75% a.a.).

### Technical

- **`materialized: table` no `dbt_project.yml` via `+materialized` é mais limpo que config inline.**
  Definir em `dbt_project.yml` aplica a todos os modelos do `domain_bcb` — adicionar um
  novo modelo não exige lembrar de incluir o bloco `{{ config(materialized='table') }}`.
  O bloco config foi mantido nos modelos mesmo assim para que cada arquivo seja autoexplicativo,
  mas o padrão correto para projetos maiores é centralizar no `dbt_project.yml`.

- **`BashOperator` herda o ambiente do container automaticamente — não usar `env=` explícito.**
  Se `env=` for usado sem incluir todo `os.environ`, o processo dbt não terá PATH, HOME,
  e outras variáveis essenciais. A forma segura é depender da injeção via `compose.airflow.yml`.

- **`ExternalTaskSensor` com `external_task_id=None` aguarda toda a DAG Bronze.**
  Sem `external_task_id`, o sensor monitora o `DagRun` completo — não uma task específica.
  Isso é o comportamento correto para cross-DAG dependency: Silver deve esperar Bronze
  completar integralmente, não apenas uma task intermediária.

- **`dbt-postgres` não tem versão fixada — possível conflito com constraints Airflow 2.10.4.**
  Assumption A-001 permanece não validada até o rebuild. Se houver conflito, fixar
  `dbt-postgres==1.8.*` que é a última versão compatível com Python 3.12 e dbt-core 1.8.x.

### Tools & Libraries

- **`EXP(SUM(LN()))` é a fórmula correta para produto encadeado em SQL — não aproximação.**
  A alternativa `PRODUCT(1 + x/100)` não existe como função nativa no PostgreSQL.
  `EXP(SUM(LN()))` é matematicamente equivalente a `∏(1 + xᵢ/100)` e é o padrão
  usado em sistemas financeiros para acumulado de retornos.

- **YAML de `schema.yml` com descrições multiline (operador `>`) é mais legível para NULLs esperados.**
  Documentar o comportamento de NULL diretamente no `schema.yml` — não apenas como comentário
  SQL — garante que `dbt docs generate` inclua a explicação na documentação do portfólio.

### Communication

- **`profiles.yml` versionado no repositório é uma escolha consciente de portfólio.**
  Em produção, `profiles.yml` ficaria fora do repo ou seria gerado dinamicamente.
  Para portfólio, commitá-lo com `env_var()` é a abordagem correta: demonstra o padrão
  de segurança (sem credenciais hardcoded) e torna o projeto reproduzível por qualquer
  recrutador que clone o repositório.
