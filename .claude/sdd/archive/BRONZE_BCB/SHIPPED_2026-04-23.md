## Summary

Pipeline de ingestão Bronze do domínio BCB entregue com DAG Airflow
`dag_bronze_bcb` orquestrando 3 tasks paralelas e independentes (SELIC, IPCA,
PTAX). A arquitetura smart first run detecta tabela vazia e executa backfill
histórico completo na primeira execução; execuções seguintes carregam apenas
o delta incremental. Idempotência garantida via `INSERT ... ON CONFLICT (date)
DO NOTHING`. Schema `bronze_bcb` isolado por domínio, alinhado com princípios
de Data Mesh. A plataforma FinLake Brasil agora tem dados macroeconômicos
brasileiros reais disponíveis para pipelines Silver/Gold e dashboards Metabase.

## Lessons Learned

### Process

- **SDD garantiu qualidade do `bcb_client.py` antes de escrever a primeira linha.**
  O Brainstorm forçou a decisão de `catchup=False` + smart first run — sem isso,
  a abordagem óbvia seria `catchup=True` com `start_date=2000-01-01`, criando
  ~9.000 runs históricos que sobrecarregariam o LocalExecutor.

- **Grounding com execução real da API foi decisivo.** Saber que o `python-bcb`
  retorna `DatetimeIndex` com `datetime64[us]` (não `date`) e que SELIC/PTAX
  pulam fins de semana evitou suposições erradas no DDL e na conversão de tipos.

- **YAGNI bem aplicado gerou 4 exclusões relevantes.** A tentação de adicionar
  Great Expectations, tabela `pipeline_runs` e Parquet paralelo é real em
  pipelines de dados. O processo formal de YAGNI ajudou a resistir sem débito
  técnico posterior.

- **API BCB SGS limita séries diárias a janelas de 10 anos.**
  `fetch_series()` precisou de chunking automático em chunks de 9 anos com
  `pd.concat()`. O DESIGN não antecipou esse limite — a assumption A-001
  (API disponível) foi validada, mas o rate/window limit não foi testado.
  Padrão a replicar para qualquer série diária futura no domínio BCB ou CVM.  

### Technical

- **`hook.return_conn()` não existe no `PostgresHook`:** O método correto é
  `conn.close()` em bloco `finally`. O DESIGN documentou `return_conn()` baseado
  em suposição — identificado e corrigido no Build. A lição: sempre verificar a
  API do hook antes de documentar padrões de código no DESIGN.

- **`TYPE_CHECKING` para imports de Airflow em módulos utilitários:** `bcb_client.py`
  usa `if TYPE_CHECKING` para o `PostgresHook`, tornando o módulo importável sem
  Airflow instalado. Padrão válido para separar utilitários de lógica de orquestração.

- **`pytest.importorskip` é o padrão correto para dependências de runtime opcionais.**
  Evita falhas de coleção no ambiente local sem perder os testes no ambiente de
  execução (container Airflow). Melhor que `@pytest.mark.skipif` porque aborta
  a importação do módulo inteiro, não apenas testes individuais.

- **`pyproject.toml` com `pythonpath = ["dags"]` resolve o problema de importação
  local sem sys.path hacks.** Replica o comportamento do Airflow que adiciona
  `dags/` ao `sys.path` — padrão moderno e limpo.

- **`AIRFLOW_CONN_FINLAKE_POSTGRES` com host `postgres:5432` (não `localhost:5433`):**
  Distinção crítica que quebra silenciosamente em runtime se ignorada. O comentário
  explícito no `.env.example` é um investimento que evitará horas de debugging
  em sessões futuras e para outros colaboradores.

### Communication

- **A pergunta sobre `start_date` do IPCA revelou um detalhe de domínio importante:**
  O dado anterior a 1994-07-01 não tem relevância analítica para séries indexadas
  ao Real. Esta decisão está documentada no ADR-002 e nos comentários da tabela.

- **Documentar o comportamento `Skipped` do IPCA no `doc_md` da DAG** evita que
  operadores interpretem o amarelo diário como falha. Observabilidade começa na
  documentação da própria DAG.

### Tools & Libraries

- **`SeriesConfig` como `@dataclass(frozen=True)`** em vez de dict simples: imutável,
  com type hints completos e accessível via atributo (não `config["key"]`). Custo zero,
  legibilidade muito melhor.

- **`_ingest_series()` genérica** reduz as 3 funções de loader públicas a 2 linhas cada.
  DRY aplicado no ponto certo: onde o padrão é idêntico e sem exceções.

- **`Makefile migrate`** com `docker exec -i` para piping de stdin: padrão correto para
  executar SQL no container sem copiar o arquivo primeiro.
